Unicode

Entrando no Unicode.

Unicode é uma sistema projetado para representar todos os caracteres de todos idiomas. Unicode representa cada letra, caractere, ou ideograma como um número de 4 bytes. Cada número representa um carácter único utilizado em pelo menos em uma das línguas do mundo. (Nem todos os números são usados, mas mais de 65.535 deles são, de modo que 2 bytes não seriam suficientes). Caracteres que são usados ​​em vários idiomas geralmente têm o mesmo número, a menos que haja uma boa razão etimológica para não. Independentemente disso, há exatamente um número por caracter, e exatamente um caracter por número. Cada número sempre significa apenas uma coisa, não há necessidade para se manter um par. U+0041 é sempre 'A', mesmo se seu idioma não tem um 'A' nele.

De cara com isto, parece ser uma boa ideia. Uma regra de codificação única à todos. Vários idiomas por documento. Não há mais "modo de comutação" para alternar entre as codificações no meio do caminho. Mas logo em seguida, a pergunta óbvia deve surgir de você. Quatro bytes? Para cada caractere ‽ </ s0> Isso parece muito desperdício, especialmente para idiomas como Inglês e Espanhol, que necessitam de menos de um byte (256 números) para expressar todos os caracteres possíveis. Na verdade, é um desperdício, mesmo para idiomas baseados em ideogramas(como o chinês), que nunca precisam de mais de dois bytes por caractere.

Há uma codificação Unicode que usa quatro bytes por caractere. Chama-se UTF-32, porque 32 bits = 4 bytes. UTF-32 é uma codificação simples, cada caractere Unicode usa um número de 4 bytes e representa o caráter com o mesmo número. Isto tem algumas vantagens, sendo a mais importante que você pode encontrar o enésimo</ v0> caractere de uma string diretamente, porque o enésimo</ v1> caractere começa no 4×N</ v2> byte. Ele também tem várias desvantagens, sendo a mais óbvia que leva quatro bytes para armazenar cada caracter.

Mesmo que haja um monte de caracteres Unicode, verifica-se que a maioria das pessoas nunca vai usar nada além dos primeiros 65535. Assim, há outra codificação Unicode, chamada UTF-16 (porque 16 bits = 2 bytes). UTF-16 codifica todos os caracteres entre 0-65535 com dois bytes, em seguida, usa alguns golpes sujos, se você realmente precisa para representar os caracteres raramente usados ​​ do "plano astral" do Unicode para além de 65535. Vantagem mais óbvia: UTF-16 usa a metade do espaço do UTF-32, porque todos os caracteres requer apenas dois bytes para armazenar, em vez de quatro bytes (exceto para os que não o fazem). E você ainda pode encontrar facilmente o enésimo</ v0> caractere de uma string diretamente, se você assumir que a string não inclui quaisquer caracter do plano astral, que é uma boa hipótese direita até o momento em que um não é.

Mas também há não-óbvias desvantagens para ambos UTF-32 e UTF-16. Diferentes sistemas de computador armazenam bytes individuais de diferentes maneiras. Isso significa que o caractere U+4E2D</ c0> poderia ser armazenado em UTF-16 como 4E 2D </ c1> ou 2D 4E</ c2>, dependendo se o sistema é big-endian ou little-endian. (Para UTF-32, existem ainda mais possibilidades de ordenamento dos bytes). Enquanto seus documentos nunca deixarem o seu computador, você estará seguro - aplicações diferentes no mesmo computador usam a mesma ordenação dos byte. Mas no minuto em que você deseja transferir documentos entre sistemas, talvez em uma rede mundial de computadores de alguma maneira, você vai precisar de uma forma para indicar que ordem os bytes estão armazenados. Caso contrário, no sistema de recebimento não tem como saber se os dois-byte da seqüência 4E 2D signfica U+4E2D ou U+2D4E</ c5>.

Para resolver este</ e0> problema, as codificações multi-byte Unicode definiem uma Marcação de ordem dos Byte("Byte Order Mark"), que é um caractere especial não imprimível que você pode incluir no início do seu documento para indicar qual a ordenação dos bytes. Para UTF-16, o Byte Order Mark é U+FEFF</ c1>. Se você receber um documento UTF-16 que começa com os bytes FF FE</ c2>, você sabe qual a ordem dos byte esta é uma maneira, se ele começa com FE FF</ c3>, você sabe que a ordenação dos bytes é invertida.

Ainda assim, UTF-16 não é exatamente ideal, especialmente se você está lidando com um monte de caracteres ASCII</ a0>. Se você pensar sobre isso, até mesmo uma página web chinesa vai conter um monte de caracteres ASCII</ A1> - todos os elementos e atributos que cercam os caracteres imprimíveis chineses. Ser capaz de encontrar o enésimo</ v2> caracter diretamente é bom, mas ainda há um problema irritante desses caracteres do plano astral, o que significa que você não pode garantir<e3/> que todo caractere tem exatamente dois bytes, então você não pode realmente</ e4> encontrar o enésimo</ v5> caractere diretamente, a menos que você mantenha um índice separado. E aí rapaz, lá com certeza há um monte de textos ASCII</ a6> no mundo ...

Outras pessoas ponderaram essas questões, e elas vieram com uma solução:

UTF-8

UTF-8 é um sistema de codificação de comprimento variável</ e0> para Unicode. É isto, diferentes caracteres levam um número diferente de bytes. Para caracteres ASCII</ A1> (az, & </ i2> c.) O UTF-8</ a3> usa apenas um byte por caractere. Na verdade, ele usa os exatos bytes; os primeiros 128 caracteres (0-127) do <A4>UTF-8</ a4> são indistinguíveis ASCII</ a5>. A extensão latina de caracteres tem o ñ e ö que usam dois bytes. (Os bytes não são o único ponto no Unicode como é o caso do UTF-16; Há alguns detalhes no arranjo dos bits) Caracteres chineses como o 中 acabam usando três bytes. Os raramente usados caracteres do "plano astral" usam quatro bytes.

Desvantagens: pois cada caracter pode ter um número diferente de bytes, encontrar o enésimo</ v0> caractere é uma operação O(N) - ou seja, quanto maior for a string, mais tempo demora para encontrar um carácter específico. Além, há um arranjo dos bits para codificar caracteres em bytes e decodificar bytes em caracteres.

Vantagens: codificação super-eficiente de caracteres ASCII comuns. Nem pior do que o UTF-16 para extesão latina de caracteres. Melhor do que o UTF-32 para caracteres chineses. Além disso (e você vai ter que confiar em mim, porque eu não estou indo mostrar-lhe a matemática), devido à natureza exata do arranjo dos bits, não há problemas na ordenação dos bytes. Um documento codificado em UTF-8 usa exatamente mesmo arranjo dos bytes em qualquer computador.

Formatando Strings

Strings podem ser defindas com aspas simples ou duplas.

Vamos dar mais uma olhada em humansize.py:

SUFFIXES = {1000: ['KB', 'MB', 'GB', 'TB', 'PB', 'EB', 'ZB', 'YB'],         ①
            1024: ['KiB', 'MiB', 'GiB', 'TiB', 'PiB', 'EiB', 'ZiB', 'YiB']}

def approximate_size(size, a_kilobyte_is_1024_bytes=True):
    '''Converter um tamanho de arquivo para uma forma legível.                          ②

    Argumentos:
    size -- tamanho do arquivo em bytes
    a_kilobyte_is_1024_bytes -- Se Verdadeiro(padrão), usa multiplos de 1024
                                Se Falso, usa multiplos de 1000

    Retorna: string

    '''                                                                     ③
    if size < 0:
        raise ValueError('o número deve ser positivo')                     ④

    multiple = 1024 if a_kilobyte_is_1024_bytes else 1000
    for suffix in SUFFIXES[multiple]:
        size /= multiple
        if size < multiple:
            return '{0:.1f} {1}'.format(size, suffix)                       ⑤

    raise ValueError('número muito grande')

  2. Funções docstrings são strings. Esta docstring ocupa diversas linhas, por isso usa três aspas consecutivas para começar e terminar a string.
  3. Estas três aspas consecutivas terminam a docstring.

Python 3 suporta formatação de valores em strings. Embora isso possa incluir expressões muito complicadas, o uso mais básico é o de inserir um valor em uma string com um único espaço reservado. 

>>> username = 'mark'
>>> password = 'PapayaWhip'                             ①
>>> "{0}'s password is {1}".format(username, password)  ②

  1. Não, minha senha não é realmente PapayaWhip.
  2. Há muita coisa acontecendo aqui. Em primeiro lugar, esta é uma chamada de método em uma string literal. Strings são objetos, e os objetos têm métodos. Em segundo lugar, toda a expressão é avaliada como uma string. Em terceiro lugar, {0} e {1} são campos de substituição, que são substituídos pelos argumentos passados ​​pelo método format().

Campos de Nomes Compostos

O exemplo anterior mostra o caso mais simples, onde os campos de substituição são simplesmente números inteiros. Campos de substituição de inteiros são tratados como índices posicionais para a lista de argumentos do método format()</ c0>. Isso significa que {0} é substituído pelo primeiro argumento (username</ v2> neste caso), {1}</ c3> é substituído pelo segundo argumento (senha</ v4>), &</ i5>c. Você pode ter tantos índices posicionais como você tem de argumentos, e você pode ter tantos argumentos como quiser. Mas os campos de substituição são muito mais poderosos do que isso. 

>>> import humansize
>>> si_suffixes = humansize.SUFFIXES[1000]      ①
>>> si_suffixes
['KB', 'MB', 'GB', 'TB', 'PB', 'EB', 'ZB', 'YB']
>>> '1000{0[0]} = 1{0[1]}'.format(si_suffixes)  ②
'1000KB = 1MB'
  1. Ao invés de chamar qualquer função no módulo humansize, você está apenas pegando uma das estruturas de dados que ele define: a lista de sufixos "SI" (potência de 1000).
  2. Isso parece complicado, mas não é. {0} remete para o primeiro argumento passado para o método format()</ c1>, si_suffixes. Mas o si_suffixes é uma lista. Então {0[0]} refere-se ao primeiro item da lista que é o primeiro argumento passado para o método format(): 'KB'. Enquanto isso, {0[1]} </ c7> refere-se ao segundo item da mesma lista: 'MB'</ c8>. Tudo fora as chaves - incluindo 1000</ c9>, o sinal de igual e os espaços - é intocável. O resultado final é a string '1000KB = 1MB'.
{0} é substituído pelo 1º argumento format(). {1} é substituído pelo 2º.

O que este exemplo mostra é que especificadores de formato podem acessar itens e propriedades de estruturas de dados usando (quase) sintaxe Python</ e0>. Isto é chamado de Campos de Nomes Compostos. Os seguintes campos de nomes compostos "apenas funciona"

  • Passando uma lista, e acessar um item da lista pelo índice (como no exemplo anterior)
  • Passando um dicionário, e acessar um valor do dicionário através de uma chave
  • Passando um módulo, e acessar suas variáveis ​​e funções pelo nome
  • Passar uma instância de classe, e acessar suas propriedades e métodos pelo nome
  • Qualquer combinação dos anteriores

Só para explodir sua cabeça, aqui está um exemplo que combina todos os itens acima: 

>>> import humansize
>>> import sys
>>> '1MB = 1000{0.modules[humansize].SUFFIXES[1000][0]}'.format(sys)
'1MB = 1000KB'

Eis como funciona:

  • O módulo sys</ c0> contém informações sobre a instância em execução Python. Desde que você acabou de importá-lo, você pode passar o módulo sys</ c1> como um argumento para o método format(). Assim, o campo de substituição {0}</ c3> refere-se ao módulo sys</ c4>.
  • sys.modules é um dicionário de todos os módulos que foram importados nesta instância Python. As chaves são os nomes de módulos como strings, os valores são os objetos do módulo em si. Assim, o campo de substituição {0.modules} refere-se ao dicionário de módulos importados.
  • sys.modules['humansize'] é o módulo humansize que você acabou de importar. O campo de substituição {0.modules[humansize]} refere-se ao módulohumansize. Note a pequena diferença de sintaxe aqui. Em um código Python real, as chaves dos dicionários sys.modules são strings, para consultá-las, você precisa colocar aspas em torno do nome do módulo (por exemplo</ i5> 'humansize '</ c6>). Mas dentro de um campo de substituição, você ignora as aspas em torno do nome da chave do dicionário (por exemplo<i7 />, humansize). Para citar PEP 3101: Formatação Avançada de String</ a9>, "As regras para analisar um item de chave é muito simples. Se começa com um dígito, em seguida, ele é tratado como um número, caso contrário, é usada como uma string. "
  • sys.modules['humansize'].SUFFIXES é o dicionário do módulo humansize. O campo de substituição {0.modules [humansize].SUFFIXES}</ c2> refere-se a esse dicionário.
  • sys.modules['humansize'].SUFFIXES[1000] é uma lista de sufixos : ['KB', 'MB', 'GB', 'TB', 'PB', 'EB', 'ZB', 'YB']. Assim o campo de substituição {0.modules[humansize].SUFFIXES[1000]} refere-se a essa lista.
  • sys.modules['humansize'].SUFFIXES[1000][0] é o primeiro item da lista de sufixos: 'KB'. Portanto, o campo substituição total {0.modules [humansize].SUFFIXES[1000][0]} é substituído pela string de dois caracteres KB</ c4>.

Formatos Específicos

Mas espere! Ainda há mais! Vamos dar mais uma olhada naquela estranha linha de código do humansize.py: 

if size < multiple:
    return '{0:.1f} {1}'.format(size, suffix)

1{1} é substituído pelo segundo argumento passado para o método format(), que é um sufixo. Mas o que é {0:.1f}? São duas coisas: {0}</ c4>, que você reconhece, e : .1f, que você não conhece. A segunda metade (incluindo e após os dois pontos) define o formato especifico</ i6>, que apura como a variável substituída deverá ser formatada.

☞ </ s0> Especificadores de formato permitem que você mude o texto substituto em uma variedade de maneiras úteis, como a função printf <d2 />() em C. Você pode adicionar zero ou espaço de preenchimento, alinhar seqüências de controle, precisão decimal, e até mesmo converter números em hexadecimal.

Dentro de um campo de substituição, dois pontos (: </ c0>) marca o início do especificador de formato. O especificador de formato ".1" significa "volta para o décimo mais próximo" (ou seja</ i2> exibir apenas um dígito após o ponto decimal). O especificador de formato "f" significa "número de ponto fixo" (em oposição à notação exponencial ou alguma outra representação decimal). Assim, dado um tamanho de 698.24 e o sufixo</ v6> 'GB', a string formatada seria '698.2 GB', porque 698.24 fica arredondado para uma casa decimal, em seguida, o sufixo é acrescentado após o número. 

>>> '{0:.1f} {1}'.format(698.24, 'GB')
'698.2 GB'

Para todos os detalhes sobre especificadores de formato, consulte a especificação do Format Specification Mini-Language na documentação oficial do Python.

  3. O método lower() converte a string toda em letras minúsculas. (Da mesma forma, o método upper()</ c1> converte uma string toda para letras maiúsculas.)
  4. O método count() conta o número de ocorrências de uma substring. Sim, realmente existem seis "f"s nessa frase!

Aqui está outro caso comum. Vamos dizer que você tem uma lista de pares chave-valor, na forma key1=value1&key2=value2, e você quer separá-los e fazer um dicionário no formato {key1: value1, key2: value2}. 

>>> query = 'user=pilgrim&database=master&password=PapayaWhip'
>>> a_list = query.split('&')                                        ①
>>> a_list
['user=pilgrim', 'database=master', 'password=PapayaWhip']
>>> a_list_of_lists = [v.split('=', 1) for v in a_list if '=' in v]  ②
>>> a_list_of_lists
[['user', 'pilgrim'], ['database', 'master'], ['password', 'PapayaWhip']]
>>> a_dict = dict(a_list_of_lists)                                   ③
>>> a_dict
{'password': 'PapayaWhip', 'user': 'pilgrim', 'database': 'master'}
  1. O método split() para divisão de uma string tem um argumento obrigatório, o delimitador. O método divide uma string em uma lista de strings com base no delimitador. Aqui, o delimitador é um caractere e comercial(&), mas poderia ser qualquer coisa.
  2. Agora, temos uma lista de strings, cada uma com uma chave, seguida por um sinal de igual, seguido por um valor. Podemos usar um list comprehensino para iterar a lista inteira e dividir cada string em duas strings com base no primeiro sinal de igual. O segundo argumento opcional do métodosplit(), é o número de vezes que deseja dividir. 1 significa "dividir apenas uma vez", para o método split() retornará dois itens da lista. (Em teoria, um valor pode conter um sinal de igual também. Se você acabou de usar 'key=value=foo'.split('='), você iria acabar com uma lista de três itens ['key', 'value', 'foo'].)
  3. Finalmente, o Python pode transformar essa lista-de-listas em um dicionário simplesmente passando-a para a função dict().

☞O exemplo anterior se parece muito com a análise de parâmetros de consulta em uma URL </ a1>, mas na vida real a análise URL<a2 /> é realmente mais complicado do que isso. Se você está lidando com parâmetros URL de uma query, é melhor você usar a função <A4>urllib.parse.parse_qs(), que trata de alguns casos extremos não-óbvios.

Fatias de Um String

Depois de definir uma string, você pode obter qualquer parte dela como uma nova string. Isso é chamado slicing(fatiar) de uma string. Fatiar strings funciona exatamente da mesma forma que fatiar listas, o que faz sentido, porque strings são apenas seqüências de caracteres. 

>>> a_string = 'Meu alfabeto comeca aonde o seu alfabeto termina.'
>>> a_string[4:12]           ①
'alfabeto'
>>> a_string[4:-4]           ②
'alfabeto comeca aonde o seu alfabeto term'
>>> a_string[0:3]            ③
'Meu'
>>> a_string[:19]            ④
'Meu alfabeto comeca'
>>> a_string[19:]            ⑤
' aonde o seu alfabeto termina.'
  1. Você pode obter uma parte de uma string, chamada de "slice", especificando dois índices. O valor de retorno é uma nova string contendo todos os caracteres da string, em ordem, começando com o índice de primeira fatia.
  2. Como cortar listas, você pode usar índices negativos para cortar strings.
  3. Strings são de base zero, assim a_string[0:2] retorna os dois primeiros itens da string, a partir de a_string[0]</ c1>, até mas não incluindoa_string[2].
  4. Se o índice da fatia esquerda é 0, você pode deixá-la fora, o 0(zero) é implícito. Assim a_string[:18]</ c0> é o mesmo que a_string[0:18] </ c1>, porque o 0(zero) está implícito.
  5. Da mesma forma, se o índice da fatia à direita é o comprimento da string, você pode deixá-lo fora. Assim a_string[18:] é o mesmo que a_string[18:44], porque a string tem 49 caracteres. Existe uma simetria agradável aqui. Nesta string de 44 caracteres, a_string[:18] retorna os primeiros 18 caracteres, e a_string[18]: retorna tudo menos os primeiros 18 caracteres. Na verdade, a_string[:n]</ c4> sempre retornará os primeiro n</ v6> caracteres, e a_string[n:]</ c7> retorna o resto, independentemente do comprimento da string.

Strings vs. Bytes

Bytes são bytes; caracteres são uma abstração. Uma seqüência imutável de caracteres Unicode é chamada de string. Uma seqüência imutável de números entre 0 e 255 é chamado de bytes</ i2> object. 

>>> by = b'abcd\x65'  ①
>>> by
b'abcde'
>>> type(by)          ②
<class 'bytes'>
>>> len(by)           ③
5
>>> by += b'\xff'     ④
>>> by
b'abcde\xff'
>>> len(by)           ⑤
6
>>> by[0]             ⑥
97
>>> by[0] = 102       ⑦
Traceback (most recent call last):
  File "<stdin>", line 1, in <module>
TypeError: 'bytes' object does not support item assignment
  1. Para definir um bytes object, use o b'' com a byte literal. Cada byte dentro do byte literal pode ser um carácter ASCII ou um número hexadecimal codificado de \x00 a \xff (0–255).
  2. O tipo de um bytes object é bytes.
  3. Assim como listas e strings, você pode obter o comprimento de alguns bytes object com a função built-in len().
  4. Assim como listas e strings, você pode usar o operador + para concatenar bytes objects. O resultado é um novo bytes object.
  5. Concatenação de um bytes</ c0> object de 5 bytes e um de 1 byte dá-lhe um bytes object de 6 bytes.
  6. Assim como listas e strings, você pode usar a notação de índice para obter bytes individuais de alguns bytes object. Os itens de uma string são strings; os itens de um bytes object são inteiros. Especificamente, inteiros entre 0-255.
  7. Um bytes object é imutável, não se pode atribuir bytes individuais. Se você precisa mudar bytes individuais, você pode usar o fatiamento de strings</ a1> e operadores de concatenação (que funcionam da mesma forma como strings), ou você pode converter os bytes object em um bytearray object. 
>>> by = b'abcd\x65'
>>> barr = bytearray(by)  ①
>>> barr
bytearray(b'abcde')
>>> len(barr)             ②
5
>>> barr[0] = 102         ③
>>> barr
bytearray(b'fbcde')
  1. Para converter um bytes object em um bytearray object mutável, use a função built-in bytearray().
  2. Todos os métodos e as operações que você pode fazer em um bytes object, você pode fazer em um bytearray object também.
  3. A única diferença é que, com o bytearray object, você pode atribuir bytes individuais usando a notação de índice. O valor atribuído deve ser um inteiro entre 0-255.

A única coisa que você nunca pode fazer é misturar bytes e strings. 

>>> by = b'd'
>>> s = 'abcde'
>>> by + s                       ①
Traceback (most recent call last):
  File "<stdin>", line 1, in <module>
TypeError: can't concat bytes to str
>>> s.count(by)                  ②
Traceback (most recent call last):
  File "<stdin>", line 1, in <module>
TypeError: Can't convert 'bytes' object to str implicitly
>>> s.count(by.decode('ascii'))  ③
1
  1. Você não pode concatenar bytes e strings. Eles são dois tipos diferentes de dados.
  2. Você não pode contar as ocorrências de bytes em uma string, porque não há bytes em uma string. Uma string é uma sequência de caracteres. Talvez você quis dizer "contar as ocorrências da string que você obteria depois de decodificação dessa seqüência de bytes em uma codificação de caracteres particular"? Bem, então você precisa dizer isso explicitamente. Python 3 não implicitamente converte bytes em strings ou strings em bytes.
  3. Por uma incrível coincidência, esta linha de código diz "contar as ocorrências da string que você obteria depois de decodificação essa seqüência de bytes nesta codificação de caracteres em particular."

E aqui está o link entre strings e bytes: bytes objects têm um método decodec() </ c1> que utiliza uma codificação de caracteres e retorna uma string, a strings tem um método encode()</ c3> que utiliza uma codificação de caracteres e retorna um bytes object. No exemplo anterior, a descodificação foi relativamente simples - a conversão de uma seqüência de bytes em uma seqüência de caracteres na codificação ASCII. Mas o mesmo processo funciona com qualquer codificação que suporta os caracteres da string - codificações mesmo legado (não-Unicode). 

>>> a_string = '深入 Python'         ①
>>> len(a_string)
9
>>> by = a_string.encode('utf-8')    ②
>>> by
b'\xe6\xb7\xb1\xe5\x85\xa5 Python'
>>> len(by)
13
>>> by = a_string.encode('gb18030')  ③
>>> by
b'\xc9\xee\xc8\xeb Python'
>>> len(by)
11
>>> by = a_string.encode('big5')     ④
>>> by
b'\xb2`\xa4J Python'
>>> len(by)
11
>>> roundtrip = by.decode('big5')    ⑤
>>> roundtrip
'深入 Python'
>>> a_string == roundtrip
True
  1. Essa é uma string. Ele tem nove caracteres.
  2. Este é um bytes object. Tem 13 bytes É a seqüência de bytes que você começa quando você toma uma a_string e codificá-la em UTF-8 .
  3. Este é um bytes object. Tem 11 bytes. É a seqüência de bytes que você começa quando você toma uma a_string </ v1> e codificá-la em GB18030.
  4. Este é um bytes object. Tem 11 bytes. É uma seqüência totalmente diferente de bytes que você começa quando você pega a_string e codifica em Big5.
  5. Essa é uma string. Ele tem nove caracteres. É uma seqüência de caracters que começa quando você a decodifica usando o algoritmo Big5 de codificação. Ele é idêntico à string original.

Postscript: Codificação de Caracteres do código fonte Python

Python 3 assume que seu código-fonte - ou seja</ 0> para cada arquivo .py -. Está codificado em UTF-8.

☞ </ s0> Em Python 2, a codificação padrãopara os arquivos . py</C2> foi ASCII. Em Python 3, a codificação padrão é UTF-8.

Se você gostaria de usar uma codificação diferente dentro de seu código Python, você pode colocar uma declaração de codificação na primeira linha de cada arquivo. Esta declaração define um arquivo .py</ c0> a ser windows-1252: 

# -*- coding: windows-1252 -*-

Tecnicamente, a substituição de codificação de caracteres também pode ser na segunda linha, se a primeira linha é um comando hash-tag do UNIX. 

#!/usr/bin/python3
# -*- coding: windows-1252 -*-

Para mais informações, consulte PEP 263: Defining Python Source Code Encodings.
